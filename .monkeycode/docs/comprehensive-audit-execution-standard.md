# 小牛育儿全面审计执行规范

**文档日期**: 2026-06-15  
**文档性质**: 审计总规范 / 执行标准 / 协作边界  
**适用对象**: 只读审查 Agent、主执行 Agent、项目负责人  
**适用范围**: `miniprogram/`、`backend/`、`.monkeycode/` 中与当前产品能力、审计记录、回测脚本、发布相关的文件与配置

---

## 一、文档目的

本规范用于定义“小牛育儿”项目全面审计工作的统一执行方法，确保审计过程具备以下特性：

1. 可执行：任何接手的 Agent 都能按同一标准完成审计。
2. 可验证：每一项结论都有对应证据、复现路径和判断标准。
3. 可交接：只读审查 Agent 与主执行 Agent 可以顺畅分工。
4. 可收口：审计结果可以直接进入修复决策与发布治理。
5. 可追责：每一类问题都能追溯到模块、责任边界、影响范围和复核结论。

---

## 二、审计总目标

本次全面审计必须完整覆盖以下六大维度：

1. 所有代码测试
2. 所有逻辑测试
3. 所有功能测试
4. 所有深度测试
5. 所有环境测试
6. 所有稳定性测试

“全面覆盖”的定义如下：

1. 每个核心模块都必须进入六维审计矩阵。
2. 每条主路径都必须覆盖正常路径、异常路径、空数据路径和边界路径。
3. 每个问题都必须给出证据、复现方法、严重级别、影响范围和修复建议。
4. 每项“未覆盖”都必须明确写出原因，不能以默认跳过处理。

## 四、角色分工

### 4.1 只读审查 Agent

职责：

1. 只读分析代码、配置、文档、脚本和测试结果。
2. 允许执行非破坏性检查命令。
3. 允许输出问题清单、证据链和风险判断。
4. 不允许修改代码。
5. 不允许修改数据库。
6. 不允许部署、重启、提交、推送。

输出要求：

1. 问题清单
2. 证据位置
3. 复现步骤
4. 风险等级
5. 是否稳定复现
6. 是否建议修复

### 4.2 主执行 Agent

职责：

1. 制定统一审计规范与修复决策标准。
2. 复核只读审查 Agent 的问题结论。
3. 判断问题是真缺陷、预期行为还是环境假阳性。
4. 根据优先级决定是否修复。
5. 对确认修复的问题负责变更、验证、留档和提交。

### 4.3 用户

职责：

1. 指定审计优先级和业务关注点。
2. 决定是否进入微信开发者工具、真机或生产回测。
3. 对高风险改动、发布窗口和实际交付节奏做最终裁定。

## 五、执行边界

### 5.1 允许的操作

1. 读代码
2. 读配置
3. 读审计文档
4. 运行语法检查、Lint、非破坏性统计脚本、只读搜索命令
5. 在工作区内新增审计文档、审计记录、回测报告
6. 在确认后执行代码修复、语法校验、回归验证、Git 提交与推送

### 5.2 禁止的操作

1. 修改 `woying-backend` 或我赢AI任何代码与配置
2. 修改无关目录中的文件
3. 删除数据、删除目录、删除库表、重置仓库历史
4. 在未确认前直接修改生产数据库数据
5. 在未确认前直接重启与本项目无关的进程
6. 输出支付密钥、证书私钥等敏感内容

### 5.3 生产环境边界

1. 所有服务器操作只允许落在小牛育儿目录和 `niuniu-backend` 进程范围内。
2. 远端生产目录不是 Git 工作树，部署必须采用最小文件同步策略。
3. 每次远端代码同步前必须有备份点。
4. 每次远端验证后必须记录验证命令、结果和影响范围。

## 六、执行前置要求

任何 Agent 在开始全面审计前，必须先完成以下动作：

1. 读取 `.monkeycode/MEMORY.md`
2. 读取相关历史审计文档和生产部署文档
3. 明确当前工作目标是“只读审查”还是“修复执行”
4. 确认工作范围仅为小牛育儿代码与文档
5. 确认当前生产 API 基址为 `https://api.woyai.cn/api/v1`
6. 确认本轮审计覆盖六大测试维度

## 七、统一问题分级标准

### 7.1 P0

定义：

1. 主路径不可用
2. 线上内容被假数据掩盖
3. 数据错乱或状态错乱
4. 登录、支付、核心内容链路中断
5. 会员判断、孩子数据、任务状态或成长记录出现错误归属

处理要求：

1. 必须优先修复
2. 必须先验证可复现路径
3. 必须补回归验证

### 7.2 P1

定义：

1. 核心功能异常但存在绕过路径
2. 年龄口径、分类口径、会员口径错误
3. 详情与列表不一致
4. 页面可进但结果被误导

### 7.3 P2

定义：

1. 边界处理异常
2. 空数据展示错误
3. 非核心路径逻辑缺陷
4. 体验明显受损但不阻断主流程

### 7.4 P3

定义：

1. 死分支
2. 不可达回退逻辑
3. 命名、结构、可维护性问题
4. 低风险代码异味

---

## 八、六类审计要求

## 8.1 代码测试

目标：确认代码本身的语法、结构、约束和可维护性达标。

必须覆盖：

1. 语法检查
2. Lint
3. 关键页面脚本逐文件检查
4. 参数处理一致性
5. 死分支与不可达分支
6. mock 回退残留
7. 旧码兼容分支是否污染主链路
8. 本地缓存与服务端结果合并逻辑
9. 代码注释与实现是否一致
10. 文案与真实业务语义是否一致

执行标准：

1. 所有关键文件必须通过语法检查。
2. 所有主链路不能存在在线模式误用本地示例的代码路径。
3. 所有参数必须有空值、类型和范围处理。
4. 所有高风险回退逻辑必须明确区分“离线演示模式”和“线上失败态”。

证据要求：

1. 文件路径
2. 行号
3. 触发条件
4. 风险说明

## 8.2 逻辑测试

目标：确认页面、接口、状态、缓存和兼容层逻辑一致。

必须覆盖：

1. 前后端字段映射
2. 年龄筛选口径
3. 分类筛选口径
4. 分页逻辑
5. 会员态与非会员态
6. 游客态与登录态
7. 有孩子与无孩子档案
8. 旧编码与新编码入口
9. 本地状态 id 与后端 canonical id 一致性
10. 列表、详情、历史、汇总数据之间的一致性

执行标准：

1. 同一业务对象在所有页面必须使用同一主标识。
2. 同一筛选条件在前后端必须语义一致。
3. 同一状态在不同入口不能出现两套结果。
4. 接口失败时必须展示真实失败结果、空态或明确引导。

## 8.3 功能测试

目标：确认用户可见功能按产品预期可用。

必须逐项验证：

1. 首页计划卡加载、点击、完成
2. 成长观察开始、作答、提交、结果、历史
3. 能力成长首页、知识点列表、知识点详情、进度提交
4. 阅读正文展示、复制和结构化字段展示
5. 育儿文章列表、筛选、详情、相关文章、收藏
6. 场景搜索热词、标签、关键词搜索、回退文章
7. 营养首页、今日推荐、热门食谱、食谱列表、详情、收藏
8. 成长记录创建、历史、汇总
9. 周总结展示与服务承接
10. 成长服务页展示、兑换码、试用、支付承接

执行标准：

1. 每个功能必须至少验证正常路径一次。
2. 每个功能必须至少验证异常路径一次。
3. 每个列表页必须至少验证空结果路径一次。
4. 每个详情页必须至少验证直开路径一次。

## 8.4 深度测试

目标：发现表面功能可用但底层逻辑存在系统性风险的问题。

必须覆盖的专项：

1. 固定 3 条、5 条、6 条示例内容伪装
2. 各年龄段内容同质化
3. 接口失败被 mock 掩盖
4. 详情页先展示示例内容再静默请求
5. 本地缓存覆盖真实内容
6. 页面标题、路由参数、详情真实字段不一致
7. 旧数据、旧编码、历史数据兼容风险
8. 数据源实际存在但筛选口径导致“看起来没内容”
9. 埋点、状态记录、收藏、完成等本地状态拆分问题

执行标准：

1. 每个主模块必须至少做一次专项排查。
2. 每个专项问题必须给出“根因层”而不是只给现象层。
3. 每个专项问题必须说明是否属于系统性问题、单点问题或历史兼容问题。

## 8.5 环境测试

目标：确认相同代码在不同环境、身份和数据状态下行为一致。

必须覆盖：

1. 本地工作区环境
2. 生产接口联调环境
3. 小程序开发者工具环境
4. 真机验证环境
5. 游客态
6. 登录态
7. 普通用户
8. 成长服务用户
9. 无孩子档案
10. 有孩子档案
11. token 过期
12. 接口 401、500、超时、空响应

执行标准：

1. 环境差异必须记录到报告中。
2. 无法在当前环境验证的项必须标记为“待外部验证”。
3. 环境限制不能被写成“通过”。

## 8.6 稳定性测试

目标：确认主流程在重复操作、快速操作和弱网条件下仍能稳定工作。

必须覆盖：

1. 重复进入页面
2. 快速切换筛选
3. 连续点击收藏
4. 连续点击完成按钮
5. 下拉刷新与分页叠加
6. 页面返回再进入
7. 缓存存在与不存在两种情况
8. 弱网或单次失败后的恢复
9. 接口偶发失败后的用户提示
10. 埋点和本地状态是否重复写入

执行标准：

1. 不能因为快速操作出现重复状态写入。
2. 不能因为重入页面导致示例内容覆盖真实内容。
3. 不能因为局部失败导致整页伪装成成功状态。

## 九、执行顺序标准

全面审计必须按以下顺序执行：

1. 先做代码与逻辑审计
2. 再做功能与深度审计
3. 再做环境与稳定性审计
4. 最后汇总问题、去重、分级、给出修复建议

推荐模块顺序：

1. 首页
2. 育儿文章
3. 营养食谱
4. 阅读详情与能力成长
5. 成长观察
6. 成长记录
7. 周总结
8. 成长服务
9. 孩子档案与个人中心

排序原则：

1. 主曝光链路优先
2. 高数据依赖链路优先
3. 曾经出现过假回退、假数据伪装的模块优先

## 十、证据标准

每一个问题条目都必须具备以下信息：

1. 问题标题
2. 模块名称
3. 页面或接口路径
4. 文件路径与行号
5. 复现步骤
6. 实际结果
7. 期望结果
8. 根因判断
9. 影响范围
10. 严重级别
11. 是否稳定复现
12. 是否建议立即修复

缺少以上字段的问题，不允许直接进入修复队列。

---

## 十一、通过标准

“审计完成”必须同时满足以下条件：

1. 六类测试都有覆盖记录。
2. 每个核心模块都有审计结论。
3. 所有 P0、P1 问题都已被识别、分级和归档。
4. 所有“未测项”都写明原因。
5. 只读 Agent 的问题清单已完成二次复核。
6. 主执行 Agent 已产出修复决策单。

“允许进入修复阶段”必须同时满足以下条件：

1. 问题已经去重。
2. 假阳性已被排除。
3. 修复边界和验证方式已明确。
4. 回归范围已定义。

---

本轮全面审计结束后，至少产出以下文档：

## 十五、修复决策规则

主执行 Agent 在接收只读审查结果后，必须按以下规则做判断：

1. 能稳定复现且影响主路径的，优先进入修复。
2. 不能稳定复现但证据明确的，进入“待复核”列表。
3. 环境导致的假阳性，标记为“环境阻塞”，不直接修代码。
4. 仅属于低风险代码异味的，集中进入收尾清理。
5. 涉及生产部署、支付、认证、成长服务、数据错归属的，必须单列为高优先级。

---

## 十六、最终要求

1. 审计过程中不允许用“看起来正常”替代证据。
2. 审计结论必须建立在代码、接口、页面表现和复现步骤之上。
3. 每个问题必须追到根因层，不能只停留在 UI 现象层。
4. 主执行 Agent 只有在完成问题复核、优先级判断和回归范围界定后，才进入修复。

本规范从创建之日起，作为本项目后续全面审计工作的统一执行标准。
