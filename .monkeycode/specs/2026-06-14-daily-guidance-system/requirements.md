# 小牛育儿日常陪伴闭环需求文档

**文档日期**: 2026-06-14  
**目标系统**: 小牛育儿小程序日常陪伴闭环  
**需求代号**: `daily-guidance-system`  
**适用范围**: 首页推荐、场景化搜索、成长记录、周总结、会员承接、运营回测

## 一、背景与目标

小牛育儿已经具备育儿内容、能力成长、营养建议、会员和 AI 问答等能力，但当前更像功能集合，离“每天都想打开的育儿助手”还有明显距离。

从宝妈视角看，最核心的问题有三类：

- 打开首页后，不知道今天先做什么。
- 遇到具体问题时，无法用最短路径拿到可执行方案。
- 连续使用一周后，看不到孩子和家庭执行的变化。

本次需求的核心目标是建立“每日建议 -> 当天执行 -> 日常记录 -> 每周总结”的使用闭环，让小程序从内容工具升级为轻陪伴型育儿助手。

本次建设必须同时满足以下原则：

- 先解决高频刚需，再扩展复杂能力。
- 首期优先复用现有内容、任务、会员、测评和埋点基础。
- 每个阶段都必须具备可回测、可回滚、可复盘能力。
- 小程序生产 API 固定使用 `https://api.woyai.cn/api/v1`。

## 二、业务目标

本期建设需要支撑以下业务结果：

1. 提升首页打开后的首屏有效点击率，让用户 30 秒内知道今天先做什么。
2. 提升带问题进入搜索时的命中率，让用户 10 秒内拿到处理建议。
3. 建立成长记录和周总结，增强连续使用和会员感知价值。
4. 为后续精细化推荐、会员转化和后台运营分析提供结构化数据基础。

建议首期业务观察指标：

- 首页今日计划卡点击率
- 今日计划完成率
- 搜索词场景命中率
- 成长记录提交率
- 周总结查看率
- 会员相关页面点击率与转化率

## 三、目标用户与使用场景

### 3.1 核心用户

#### 宝妈用户

- 希望快速得到“今天该做什么”的明确建议。
- 希望问题出现时有直接可执行的方法。
- 希望看到孩子近期有没有进步，自己的陪伴有没有效果。

#### 家庭协同照护者

- 需要快速了解当前孩子的重点关注事项。
- 需要跟随统一建议执行，减少家庭成员之间的做法冲突。

### 3.2 典型使用场景

1. 早晨打开小程序，查看今日重点陪伴建议。
2. 晚饭时遇到孩子不吃饭，直接搜索“孩子不吃饭怎么办”。
3. 晚上睡前花 1 分钟记录当天饮食、睡眠、情绪与任务完成情况。
4. 周末查看本周成长总结，了解改善点和下周建议。

## 四、建设范围

### 4.1 本期范围

本期正式建设以下四个模块：

1. `今日育儿计划`
2. `场景化搜索`
3. `成长记录`
4. `每周成长总结`

同时建设以下配套内容：

- 会员承接与权益表达调整
- 推荐规则与场景词库初版
- 埋点补全
- 回测与发布要求

### 4.2 本期不纳入范围

以下能力留待后续版本：

- AI 自动生成复杂个性化周报
- 家庭多成员协同编辑记录
- 长周期医学趋势分析
- 后台可视化规则编排器
- 全量内容编辑后台重构

## 五、核心需求

### 5.1 今日育儿计划

#### 5.1.1 目标

首页首屏必须给出当天最值得执行的建议，减少用户选择成本。

#### 5.1.2 展示要求

系统必须支持：

- 首页展示 `3` 张今日计划卡。
- 每张卡展示标题、建议原因、建议动作、预计时长和入口按钮。
- 支持至少以下卡片类型：
  - 能力训练卡
  - 营养建议卡
  - 习惯提醒卡
- 展示当天已完成数量，例如 `1/3`。

系统应支持：

- 根据孩子年龄、测评结果、最近行为和最近内容消费情况调整推荐顺序。
- 对新用户展示默认新手计划。
- 对无孩子档案用户展示补充资料引导。

#### 5.1.3 规则要求

首期推荐规则必须满足：

- 优先按孩子年龄段过滤可用内容。
- 对最近测评中的低分维度进行加权。
- 对最近 7 天低活跃的关键板块进行补足推荐。
- 每日最多推荐 `3` 张卡。
- 同类型卡片每日最多 `2` 张。

#### 5.1.4 完成要求

系统必须支持：

- 用户点击计划卡进入目标页面。
- 用户完成后回写完成状态。
- 完成记录用于周总结和后续推荐。

#### 5.1.5 验收标准

- 不同年龄孩子可以拿到不同计划。
- 新老用户都能拿到有效计划数据。
- 计划完成状态刷新正确。
- 页面无数据时有明确兜底文案与入口。

### 5.2 场景化搜索

#### 5.2.1 目标

用户输入真实育儿问题时，首屏优先看到处理方案，而不是只看到文章列表。

#### 5.2.2 展示要求

搜索页必须支持：

- 顶部展示常见问题快捷入口。
- 顶部展示场景分类入口：
  - 喂养吃饭
  - 睡眠作息
  - 情绪行为
  - 习惯养成
  - 认知学习
  - 社交表达
- 结果页优先展示方案卡，再展示任务卡、文章卡和营养建议卡。

#### 5.2.3 命中要求

系统必须支持：

- 用户搜索词映射到标准场景标签。
- 每个场景标签至少关联：
  - 一句处理原则
  - 一条推荐动作
  - 一项推荐任务或文章
  - 一项相关营养或习惯建议
- 结果支持年龄段过滤。

系统应支持：

- 搜索别名和口语化表达归一化。
- 热门场景词优先返回结构化方案。

#### 5.2.4 验收标准

- 首批高频问题可稳定命中标准场景。
- 首屏方案卡可直接打开目标内容。
- 结果页不存在只返回空列表的硬失败体验。
- 搜索过程全链路埋点完整。

### 5.3 成长记录

#### 5.3.1 目标

让家长用最低成本记录当天关键状态，为周总结和长期留存提供依据。

#### 5.3.2 记录要求

系统必须支持：

- 按孩子维度记录每天状态。
- 单日记录至少覆盖以下字段：
  - 饮食情况
  - 睡眠情况
  - 情绪情况
  - 任务完成情况
  - 排便情况
  - 家长备注
- 每日每个孩子保留 `1` 条汇总记录，可反复修改。
- 多孩子切换。

系统应支持：

- 使用选项型录入，减少手动输入。
- 首次进入时显示填写引导。
- 历史记录按周查看。

#### 5.3.3 体验要求

- 完成单日记录的操作路径控制在 `1` 分钟内。
- 记录页优先展示快捷选项，再展示备注输入。
- 记录成功后给出轻反馈。

#### 5.3.4 验收标准

- 单日记录可新增、可修改、可回显。
- 切换孩子后数据独立。
- 历史页可按日期读取。
- 空数据时有补录引导。

### 5.4 每周成长总结

#### 5.4.1 目标

让家长感受到一周陪伴的成果和下周重点，强化产品陪伴价值。

#### 5.4.2 展示结构

系统必须支持生成以下内容：

- 本周整体状态
- 做得好的地方
- 需要继续关注的地方
- 下周建议 `3` 条
- 推荐内容 `2` 条

#### 5.4.3 生成要求

首期总结必须基于规则生成，输入来源至少包括：

- 成长记录趋势
- 今日计划完成率
- 最近能力任务完成情况
- 最近浏览内容
- 最近测评维度

系统应支持：

- 普通用户查看基础版周总结。
- 会员查看完整建议版周总结。

#### 5.4.4 验收标准

- 同一周内重复读取结果一致。
- 总结文案与真实记录数据一致。
- 无记录用户可得到轻量版引导总结。

### 5.5 会员承接

会员相关页面与触点必须明确表达以下权益：

- 每日个性化育儿计划
- 每周成长总结
- 更完整的成长趋势与建议
- 专属内容与礼包权益

系统应确保：

- 会员权益文案与实际开放功能一致。
- 限制能力使用时给出明确升级提示。
- 不影响现有统一兑换码与会员发放逻辑。

## 六、内容与规则要求

### 6.1 推荐内容来源

首期建议复用以下现有能力：

- 育儿文章
- 能力成长任务
- 营养健康内容
- 测评结果
- 用户最近行为

### 6.2 内容组织要求

- 今日计划和周总结都必须输出“先做什么”的结论。
- 搜索方案卡必须优先输出一句处理原则和一条建议动作。
- 所有推荐内容都必须支持年龄段和主题口径一致。

## 七、埋点与数据要求

本期必须新增或完善以下事件：

- `daily_plan_view`
- `daily_plan_click`
- `daily_plan_complete`
- `scene_search_exposure`
- `scene_search_submit`
- `scene_solution_click`
- `growth_record_open`
- `growth_record_submit`
- `weekly_summary_view`
- `weekly_summary_action_click`

埋点必须至少包含以下上下文：

- `child_id`
- `membership_type`
- `module_key`
- `content_type`
- `content_id`
- `scene_key`
- `plan_type`
- `record_date`

## 八、非功能要求

### 8.1 性能

- 首页今日计划接口首屏建议控制在 `500ms` 至 `1200ms` 范围内。
- 搜索结果接口应优先返回结构化结果，避免长时间空白等待。
- 周总结生成必须支持缓存，避免重复请求反复计算。

### 8.2 一致性

- 小程序首页、搜索、详情、记录和会员触点口径必须统一。
- 年龄段命名必须与现有育儿内容体系一致。

### 8.3 安全与隔离

- 所有接口继续走小牛育儿现有 API 域名与认证体系。
- 服务端改动只允许落在小牛育儿目录与 `niuniu-backend` 进程范围内。
- 不得影响 `我赢AI` 的目录、路由、数据库和 `woying-backend` 进程。

## 九、分阶段交付要求

### 9.1 开发顺序

建议按以下顺序推进：

1. 今日育儿计划
2. 成长记录
3. 每周成长总结
4. 场景化搜索
5. 会员承接优化

### 9.2 每阶段交付要求

每阶段都必须包含：

- 变更说明
- 数据结构或接口变更说明
- 影响范围说明
- 回测清单
- 回滚方式
- Git 提交与推送记录

## 十、回测与发布要求

### 10.1 通用要求

每轮开发完成后必须执行回测，且回测记录需要包含：

- 回测时间
- 回测版本或提交号
- 回测范围
- 回测方法
- 实际结果
- 是否允许进入下一阶段

### 10.2 模块回测要求

#### 今日育儿计划回测

- 新用户、有孩子档案用户、会员用户三类账号都要验证。
- 不同年龄段孩子至少验证 `3` 组样本。
- 验证计划展示、点击、完成、刷新状态。
- 验证无数据兜底与弱网重试。

#### 场景化搜索回测

- 首批高频问题词至少回测 `30` 个。
- 验证别名词、口语词和年龄过滤。
- 验证方案卡、任务卡、文章卡的跳转路径。
- 验证空结果和低命中场景的兜底文案。

#### 成长记录回测

- 单日新增、修改、回显、切换孩子、历史查看全部要验证。
- 验证连续 7 天记录后的趋势读取。
- 验证未登录、无孩子、无记录三类边界状态。

#### 每周成长总结回测

- 验证有记录用户、弱记录用户、无记录用户三类样本。
- 验证普通用户与会员用户看到的内容差异。
- 验证总结结论与原始记录一致。

### 10.3 上线前要求

上线前必须完成：

- 接口自测通过
- 小程序开发者工具完整流程回测通过
- 至少一轮真机场景回测通过
- 埋点事件核对通过
- 生产变更备份完成
- 回滚方案确认完成

## 十一、风险与约束

当前主要风险如下：

- 推荐规则过重会拖慢首期落地。
- 场景词库质量直接决定搜索体验。
- 成长记录流程过长会显著拉低提交率。
- 周总结文案如果空泛，会损伤用户信任。

对应约束如下：

- 首期优先规则化方案，不引入重型模型依赖。
- 先覆盖高频核心场景，再扩展长尾问题。
- 记录流程坚持一屏快填优先。
- 总结文案坚持“数据结论 + 下一步建议”结构。

## 十二、需求验收结论

只有同时满足以下条件，才能认为本需求进入可上线状态：

1. 四个核心模块都已具备用户可感知能力。
2. 首页、搜索、记录、周总结形成完整闭环。
3. 核心回测用例全部通过并留档。
4. 埋点、备份、回滚方案齐备。
5. 生产验证未影响 `我赢AI` 相关服务。
